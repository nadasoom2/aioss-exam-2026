# 브랜치 전략 선언

이 프로젝트는 **Trunk-Based Development (TBD)** 전략을 사용합니다.

모든 개발자는 `main` 브랜치를 기준으로 단명(short-lived) 기능 브랜치를 생성하며, 작업 완료 즉시 PR을 통해 `main`에 머지합니다.

브랜치는 1개 문항(기능) 단위로 생성하고, 머지 후 즉시 삭제하여 브랜치 수명을 최소화합니다.

대규모 변경은 Feature Flag를 활용해 미완성 코드도 안전하게 `main`에 통합할 수 있도록 합니다.

모든 PR은 리뷰 체크리스트를 통과해야 하며, CI 파이프라인이 통과된 경우에만 머지를 허용합니다.

## 브랜치 네이밍 규칙

| 유형 | 형식 | 예시 |
|------|------|------|
| 기능 개발 | `feature/<설명>` | `feature/workflow-setup` |
| 버그 수정 | `fix/<설명>` | `fix/ci-timeout` |
| 문서 작업 | `docs/<설명>` | `docs/readme-update` |

## 머지 규칙

- `main` 브랜치에 직접 push 금지
- PR은 최소 리뷰 코멘트 1개 이상 필요
- CI 파이프라인 통과 필수
